React:Render()

  • Mixins
  • React
  • Render

用React写页面的时候，经常会采用嵌套组件的结构，不同的组件间会有父子关系。

简单而言，在组件A的Render函数中如果出现了另一个组件B，则A的这个实例是B实例的父组件。

React构建的页面是基于单向数据流，不像Angular那样做双向绑定，所以数据的流向只能单向从父—>子（设置子组件的props，类似传参），也可以子—>父（给子组件传入自身函数的引用，类似回调）。

但是在实际做时，一旦关联了父子数据，我们会发现React的Render方法有点鬼畜。

参考此文：understanding-react-js-rendering

我们的jsx中可能会经常这么写：

Father中会有若干个Child实例作为子组件，Father的函数changeState作为参数传入Child.
而在Child处理完自身的点击事件（可能是样式修改）后，会利用props获取的引用，调用父组件的方法，改变父组件的状态。

疯狂的Render

看起来很完美，但是我们会发现之前对子组件做的样式修改是无效的。

实际上，在我们修改父组件的状态时，不仅父组件的render方法被调用，所有子组件的render也会被调用，又被渲染回最初的状态。子组件自身的状态修改不会触发父组件的render.

如果一个app中有五层的组件嵌套，那么在最顶层父组件render时，下面的子组件也要render，其中最底层会render五次。

这听起来挺影响性能，官网的解释是前端的性能瓶颈在于dom操作，而不是js的执行速度。

When does React Render Components?

总结一下Render触发的情况：

  1. 页面上挂载组件时
  2. 本身状态发生改变，即this.setState({})被调用
  3. 上层的状态发生改变
  4. 上层传入的props发生改变
