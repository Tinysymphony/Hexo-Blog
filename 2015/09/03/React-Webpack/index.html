React+Webpack

  • React.js
  • atom
  • scss
  • webpack

学习各种前端姿势时，感觉前端正处于风起云涌的时代。各种框架、js方言、XXX.js不断涌现，拥有大公司背景的技术也挡不住时代的潮流，或是作古或是推陈出新割据一方。

最近是在逛各种博客，感觉应该接触一些框架或者说设计思想，感受前端的哲学以及技术变迁，而不是赶场凑热闹、把自己束缚在人家设置的条条框框里。

上个星期在看React的内容，很干净的View层。在初学前端的时候感觉分割Html和js是很自然很本能的事，但接触React之后感觉混合也不错，一个个组件很清晰地被组装起来，方便管理。

看了两篇文章，都提到了webpack作为前端资源的管理器。所以我也做了一点尝试，准备用React+Webpack开发一个web应用，用于分享半音阶简谱和歌词。

先贴两篇文章
1. infoQ某文 (代码有小问题，大致内容易懂)
2. 轻松入门React和Webpack

Webpack更详细的资料建议移步webpack cool book

写下我的开发环境搭建过程：

npm init #略
npm install --save react
npm install -g webpack webpack-dev-server #后者是一个基于express的http开发用服务器
npm install --save-dev jsx-loader css-loader style-loader url-loader file-loader #各种loader是webpack整合资源时使用的工具,还可以添加scss，less等东西
npm install --save-dev react-hot-loader #支持修改文件后自动更新模块，方便开发

webpack的哲学是根据配置文件管理所有的web资源（js代码、js方言代码、模板、图片、样式等），寻找依赖，生成相应的打包文件（pack），高效开发。

webpack需要在项目文件下写自己的配置文件webpack.config.js，也可以在使用时通过—config参数指定，如果加上-p参数则会压缩混淆文件，在发布应用时才会使用。

目前我的webpack.config.js内容如下：

entry是输入文件，一般要填写入口文件的位置（相对config的路径），之前的两条webpack*是为了让webpack-dev-server配合react-hot-load

传入路径字符串: entry: './js/index.js
使用字符串数组: entry: ['a.js', 'b.js']
使用对象: entry: { a: ['a1.js', 'a2.js', 'a3.js'], b: 'b.js' }

output对应输出，我这里的写法filename: [name].bundle.js是指根据entry传入的属性名生成多个文件，比如上面那条规则，是整合解析a1.js/a2.js/a3.js生成a.bundle.js，并且解析b.js生成b.bundle.js
文件在当前目录的assets文件夹下，之后只要在html中加入xxx.bundle.js就能实现相应功能。

model里面的loaders指定上面所说的解析步骤需要的解释器，有点类似指定编译器。因为使用jade、jsx等模板或者是scss、less等语法时需要在上线前转化成标准的js和css，这个步骤webpack在打包时帮我们做。这里的loader都是需要用npm安装一下的。

test内容匹配文件名，使用正则，exclude排除目录下一些不需要使用的文件，loaer或者是loaders指定一个或多个解释器，分别需要传字符串和字符串数组。
处理过程是从右到左的，假设我们匹配到了scss文件，loader:'style-loader!css-loader!sass-loader'
指定处理过程是：sass-loader -> css-loader -> style-loader
其实在指定的时候，’-loader’是可以忽略的，可写成loader: 'style!css!sass'

resolve 里面添加extension，是用于识别扩展名，第一个’’空字符串可以在无后缀时匹配文件名。

plugins添加插件数组，用的热插拔插件和免报错，也可以自己定制。

实际上在了解webpack的基本配置方式后，我还是花了一点时间才完全理解并配置成功的。主要是没有理解这个流程。
假设entry里面先忽略热插拔插件，单纯的写entry: {index: './js/index.js'}

在调用webpack命令时，webpack去查找目录下的index.js文件，读取里面的内容，再去寻找内容里提到的其他依赖。
这里的依赖你可以写React组件声明的js，甚至可以写css/scss/less，还可以写图片资源！

如下是我的入口文件

我的入口文件和头文件一样，先是引入了样式（这里使用scss预处理器），其实在require过程中可以指定loader，例子在注释里。
再是引入React.js文件，该文件中require了其他react组件模块，并且渲染页面。
最后引入primodial.js文件，里面是原生js，主要做一些基础的页面操作。

所以html里没必要注明css文件位置，webpack会在xxx.bundle.js被调用时在head里面插入css，或是插入由scss编译而成的css
如果有图片的话（暂时没试过），也会被引用。
这样所有的资源可以在入口js文件里面声明，webpack辅助管理。

对于React，这个入口文件一般写渲染(render)指令。
还有需要注意的是，各个组件js需要用module.exports导出，遵循CommonJS规范。

Webpack-dev-server

照搬人家写的服务器启动文件，注意这里是要和webpack联动的，所以要指定配置文件，启动第一个参数就是config文件初始化的webpack实例，第二个配置对象里指定了输出目录（其实可以忽略），hot设为true支持热插拔，后两个暂时不明。

开个终端，在项目目录下node app.js

目录结构如下
命令：tree . -I 'node_modules|*.log'

之后如果资源文件有修改并保存后，webpack会自动运行，更新bundle.js内容，浏览器只要刷新一下就行。
避免了重启服务器、或是每次刷新浏览器要等待解析scss/less/jsx/jade/coffee-js等东西的麻烦。

注：css是不用刷新浏览器动态更新的，
在浏览器中访问http://localhost:your-port/webpack-dev-server/bundle
可以看到开发调试用的页面，修改css文件保存后自动更新内容。

浏览器中看到如下log，配置成功。

Atom

在开发时主要用atom编辑器，虽然有时候卡了点，但很合我这种外貌协会人员的胃口，并且插件也很多。
需要插件时，可以直接在命令行下使用apm包管理器安装。
比如：apm install react安装react语法支持和高亮。

  • 自npm以来，类似的包管理：bower、apm、*pm还是挺好用的。

Atom小bug？

尝试在入口文件添加css文件再使用webpack时，可以看到bundle文件中是这样引入css的：

exports.push([module.id, “一个长字符串，内容是css文件”，””]);

但在atom中查看的时候从长字符串开始到后面所有js代码都变成了字符串的高亮。
原本以为是css文件里面的内容截断了长字符串，但检查后没有发现双引号存在，并且语法也是正确的。
再看长字符串内容也是没有截断，尝试着运行也能正确显示样式，后面的js代码也能正常工作。

用sublime查看代码是正常的。
所以怎么想都是atom的锅。
